## 10. 第 16 轮建议如何转成后台数据质量检查功能

### 10.1 建议新增“数据质量检查”后台页

| 检查对象 | 自动检查规则 | 风险级别 |
|---|---|---|
| 首页视频 | `enabled=true` 时必须有 `videoUrl`；建议有 poster | 中 |
| 首页 12 图 | 启用槽位必须有 `mediaUrl` 和 `alt` | 高 |
| 文章 | published 文章必须有 title、slug、summary、content、seoTitle、seoDescription、keywords | 高 |
| 文章 | content 包含“待补充”或正文过短时标红 | 高 |
| 文章 | FAQ 为空时提示 | 中 |
| 案例 | published 案例必须有 title、slug、summary、contentText、seoTitle、seoDescription | 高 |
| 案例 | published 案例缺封面、缺图片、缺 eventDate、地点含“待确认”时标红 | 高 |
| 场景 | 场景必须保留 7 个默认 slug | 高 |
| 场景组 | enabled=true 时普通组至少 1 张图，视频数字资产组至少 1 个素材 | 高 |
| 媒体 seed | seed 条目未导入媒体库时提示 | 中 |
| 媒体 | active 图片建议必须有 alt；二维码建议标注确认状态 | 中 |

### 10.2 建议输出三类结果

| 类型 | 含义 | 示例 |
|---|---|---|
| 阻塞发布 | 不处理不应静态发布 | 二选一占位正文、案例缺公开权限、启用场景组无素材 |
| 发布前建议 | 不一定阻塞，但影响 GEO 质量 | 缺 FAQ、缺图片 alt、SEO 描述过短 |
| 后续优化 | 不影响当前迁移闭环 | logo 分类、关键词微调、视觉优化 |

### 10.3 建议第 16 轮执行顺序

1. 先做只读质量检查 API，读取现有 JSON 并返回问题列表。
2. 后台新增“数据质量检查”页面，按模块展示高 / 中 / 低优先级。
3. 给每条问题提供跳转入口，例如跳到文章编辑、案例编辑、场景组编辑。
4. 增加发布前门禁规则：有高优先级问题时不建议进入静态发布。
5. 暂不自动修复数据，避免误改人工内容。
